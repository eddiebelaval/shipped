**Shipped. Daily, Friday, July 3, 2026**

Anthropic renamed Claude Code's default permission mode to say what it actually does — Manual — and stopped Claude from answering its own questions. OpenAI's coding agent defaults the other way. The gap between those two factory settings is the most interesting product argument on the frontier this week.

## Lead — the default now waits for you

Claude Code v2.1.200 landed July 3 at 16:52 UTC ([releases](https://github.com/anthropics/claude-code/releases)) with two changes that read as one decision. The mode formerly labeled "default" is now called "Manual" across the CLI, VS Code, and JetBrains — same slot, honest name, with `--permission-mode manual` and `"defaultMode": "manual"` accepted alongside the old `default`. And AskUserQuestion dialogs no longer auto-continue. When Claude asked you something mid-session and you walked away, it used to pick its own recommended answer after an idle timeout and keep working. Now it waits. Indefinitely, unless you opt back into the timeout via `/config`.

The mechanism is small. The posture is not. A question that answers itself after a timeout is a suggestion with a deadline; as of v2.1.200, a question is a blocking call — the session parks until a human shows up. If you run long sessions and count on Claude to self-resolve its own check-ins, your workflow changed this afternoon: runs that used to finish overnight will be sitting at a prompt in the morning. That's the blast radius, and it's the good kind — visible, immediate, and located exactly where the tool was previously guessing on your behalf.

Read it against the other lane. OpenAI's Codex CLI defaults to Auto: read, edit, and run commands inside the workspace without asking, prompting only at the workspace boundary or for network access ([Codex docs](https://developers.openai.com/codex/agent-approvals-security)). Same dial, opposite default. Anthropic is pricing a wrong guess as more expensive than an interruption; OpenAI is pricing interruption as the thing that kills adoption. Both positions are defensible. Only one of them assumes the agent is right when nobody is watching.

**The read:** the agent labs spent the last year racing to take the human out of the loop. The first thing Anthropic shipped in July is a default that puts the human back in.

**The move:** update, then choose on purpose. `/config` re-enables the idle timeout if you liked the self-driving check-ins; `--permission-mode manual` makes the new name explicit in scripts before the old alias fades.

## The Dig

### The daemon was killing its own agents

v2.1.199 (July 2, 23:35 UTC) and v2.1.200 between them carry a dozen background-agent daemon fixes, and one of them is a horror story: after an unclean shutdown left a corrupted worker record, the Linux daemon would kill itself — and every running agent with it — roughly every 50 seconds ([releases](https://github.com/anthropics/claude-code/releases)). v2.1.200 fills out the genre: a stale `daemon.lock` whose PID the OS had reused, so agents never started again after a crash; a handover protocol that let a reinstalled older build seize the daemon from a newer one, now settled by the version's embedded build timestamp; socket auth tokens quietly stripped from the roster during restarts.

If you ran background agents on Linux this week and watched them die at random, that was not your machine. A 50-second kill loop is precisely the failure you burn an evening blaming on your own box — out of memory? systemd? — because "the daemon is executing its children on a timer" is not a hypothesis sane people reach for first.

The pattern is the story. Lock files, PID reuse, handover rules, token persistence: these are the problems cron and sshd solved decades ago, and they are surfacing now because background agents crossed the line from feature to infrastructure. Daemons age in dog years, and Anthropic's is teething in public at two releases a day. **The move:** update to v2.1.200 before you debug anything agent-shaped on Linux tonight.
